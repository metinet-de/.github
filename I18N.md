# Internationalization (i18n) Architecture

## Overview

metinet.de supports **German** and **English**. The current routing is mixed by design: the root homepage `/` is German, the explicit English homepage lives at `/en/`, English section pages such as `/blog/` and `/contact/` stay at the root, and German equivalents live under `/de/`.

## URL Structure

| Content | Current URL | Example |
|---|---|---|
| German homepage | `/` | `/` |
| English homepage | `/en/` | `/en/` |
| English section pages | root | `/blog/`, `/contact/` |
| German section pages | `/de/` | `/de/blog/`, `/de/contact/` |
| English blog posts | `/blog/` | `/blog/2026/03/07/clean-code-was-always-for-humans/` |
| German blog posts | `/de/blog/` | `/de/blog/2026/03/07/clean-code-war-schon-immer-fuer-menschen/` |

## Key Concepts

### 1. `lang` Front Matter

Every page and post **must** have a `lang` key:

```yaml
lang: de   # or "en"
```

The `_config.yml` defaults set `lang: de` globally. Pages and posts that render in English must explicitly set `lang: en` in their front matter.

### 2. `ref` Front Matter (Translation Linking)

The `ref` key is a shared identifier that links equivalent content across languages:

```yaml
# English post
lang: en
ref: future-software-dev-ai

# German translation
lang: de
ref: future-software-dev-ai
```

Pages and posts with the same `ref` are considered translations of each other. The language switcher and translation banner use this to find counterparts.

The current root homepage `/` is the notable exception: it is German-only in front matter and relies on the language switcher's homepage fallback instead of a shared `ref`.

### 3. UI Translations

All translatable UI strings live in:

- `_data/i18n/en.yml` — English
- `_data/i18n/de.yml` — German

Access in templates:

```liquid
{% assign current_lang = page.lang | default: "de" %}
{% assign t = site.data.i18n[current_lang] %}
{{ t.nav.home }}
```

### 4. Includes

| Include | Purpose |
|---|---|
| `_includes/lang-switcher.html` | Language toggle link in the navigation header |
| `_includes/translation-banner.html` | Banner on posts linking to the translation counterpart |

## Adding a New Language

1. Create `_data/i18n/<code>.yml` with all translation keys.
2. Create `<code>/` directory with `index.html`, `blog/index.md`, `contact.html`, etc. if the language should have explicit prefixed routes.
3. Add defaults in `_config.yml` under the new path.
4. Update `tailwind.config.js` content paths.
5. Update `lang-switcher.html` to handle the new language.

## Adding a New German Blog Post

1. Create the file in `_posts/` with the format `YYYY-MM-DD-slug.md`.
2. Set front matter:

```yaml
---
layout: post
title: "Deutscher Titel"
date: YYYY-MM-DD HH:MM:SS +0100
categories: [...]
tags: [...]
author: Metin Özkan
description: "..."
lang: de
ref: shared-ref-key
permalink: /de/blog/:year/:month/:day/:title/
---
```

3. The `ref` must match the English counterpart's `ref` (if one exists).
4. The `permalink` must include the `/de/` prefix.

## Adding a New English Blog Post

1. Create the file in `_posts/` with the format `YYYY-MM-DD-slug.md`.
2. Set front matter:

```yaml
---
layout: post
title: "English Title"
date: YYYY-MM-DD HH:MM:SS +0100
categories: [...]
tags: [...]
author: Metin Özkan
description: "..."
lang: en
ref: shared-ref-key
---
```

3. The `ref` must match the German counterpart's `ref` (if one exists).
4. English posts currently render under `/blog/`.

## How the Language Switcher Works

1. Reads `page.lang` and `page.ref`.
2. Searches `site.pages` and `site.posts` for a matching `ref` in the other language.
3. If found, links to that page/post.
4. If not found, falls back to the other language's home page.

## SEO

- `<html lang="...">` is set from `page.lang` in `default.html`.
- `<link rel="alternate" hreflang="...">` tags are generated automatically for pages with a `ref` that has a counterpart.
